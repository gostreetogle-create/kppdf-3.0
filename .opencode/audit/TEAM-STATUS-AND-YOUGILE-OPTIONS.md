## 1. Контекст и боль

### Что произошло

В ходе работы над KPPDF 3.0 накопилось ощущение, что **инфраструктура управления проектом «не работает как задумано»**:

1. **UI-фиксы «невидимы» в трекере.** Вчера закрыли десятки пунктов по кнопкам, kp-button, QuotationEditor — в YouGile на доске «тишина», потому что эти работы не заведены карточками и не попадают в PLM-дерево.
2. **Дублирование источников правды.** Одновременно существуют:
   - markdown-файлы в `.opencode/audit/` (backlog, UI-план);
   - доска «Статус реализации» в YouGile (PLM-109…PLM-142, EPIC, %);
   - `.opencode/project-readiness.yaml` (проценты модулей, `enabled: false`);
   - локальный `yougile-sync-server` (порт 3002, sibling-репозиторий).
3. **Markdown «лежит в репо и врёт».** Файлы вроде `UI-CONSISTENCY-PLAN.md` содержат детальный план P0–P2, но **статус выполнения** в них не синхронизирован с тем, что видит человек в YouGile. Отсюда фрустрация: «столько кода перелопатилось — а где это отражено?»

### Что уже есть в проекте (не выбрасыать — понять роли)

| Слой | Расположение | Задуманное назначение |
|------|--------------|----------------------|
| **AI / код-инварианты** | `.opencode/` — `AGENTS.md`, `AI_CONTEXT.md`, `lock/`, `rules/`, `audit/` | Правила для агентов, FreezeGuard, чеклисты качества, backlog для AI |
| **YouGile (облако)** | https://yougile.com — доска «Статус реализации», реестр PLM-xxx | Человекочитаемая дорожная карта PLM, EPIC, % готовности |
| **yougile-sync-server** | Sibling-репо: `../yougile-sync-server` (не внутри kppdf-3.0) | Polling API YouGile → обновление % в описаниях EPIC → Google Sheets |
| **tools/** | `yougile-sync.ps1`, `yougile-registry.js`, `sync-plm-checklist.js`, `setup-documents-yougile-module.js` | Ручные и полуавтоматические операции с YouGile API |
| **Документация интеграции** | `docs/integrations/yougile/` | Конвенции, API, adding-tasks |
| **Реестр PLM → UUID** | `config/yougile-task-registry.yaml` | Связь hash-ссылок `#PLM-126` с UUID задач |

**Вывод:** это не «мусор», а **недостроенная интеграция** — каждый слой создавался под свою роль, но **мост между «человеком в YouGile» и «AI в репо» не довели до конца**.

### 2.2. Что реально делает yougile-sync-server

| Параметр | Значение |
|----------|----------|
| **Расположение** | Отдельный репозиторий рядом с kppdf-3.0: `D:\invSportiN\Сайт\yougile-sync-server` (путь зависит от машины) |
| **Запуск** | `start.ps1` поднимает его вместе с backend/frontend **локально**; health: `http://localhost:3002/api/health` |
| **Направление** | **One-way:** YouGile API → сервер → обновление описаний задач + Google Sheets |
| **Механизм** | Polling каждые ~5 мин (`pollIntervalSec`); читает `config/mapping.yaml` (модуль → задача YouGile) |
| **Что считает** | % выполнения подзадач EPIC; пишет в описание строки вида `🔵 db-models — готовность: 67%` |
| **Что НЕ делает** | Не читает `.opencode/audit/*.md`; не создаёт карточки; не синхронизирует backlog; не двусторонний sync |
| **Зависимости** | `YG_TOKEN` / `YG_API_TOKEN` в `.env` sibling-репо; без токена и запущенного процесса % не обновляются |
| **Из дома** | ❌ Требует локально запущенный dev-стек или отдельный хостинг sync-server |

### 2.3. Что НЕ синхронизировано (и не будет без решения команды)

- `CHECKLIST-BACKLOG.md` — сессионный backlog, QuotationEditor, деплой, e2e
- `UI-CONSISTENCY-PLAN.md` — 28+ задач миграции UI-кита (P0/P1/P2)
- FreezeGuard — статусы frozen/locked/wip **намеренно** только в репо (защита кода, не задачи)
- `project-readiness.yaml` — file-only tracking, UI «Статус реализации» снят с продакшена
- Новые UI-фиксы без PLM-id — не попадают никуда, кроме git history

### 2.4. Частично работающие ручные инструменты

```bash
# Показать задачу по PLM-id
node tools/yougile-registry.js show PLM-126

# Обновить описание + чеклист одной карточки после фикса в коде
node tools/sync-plm-checklist.js PLM-139 --done

# PowerShell: отчёт / статус / mark-done
pwsh tools/yougile-sync.ps1 -Action status
```

Эти скрипты **точечные** — покрывают отдельные PLM-id из реестра, не весь backlog.

## 3. Зачем создавались слои (назначение, не мусор)

### `.opencode/` — мозг для AI и защита кода

- **AGENTS.md** — архитектурные инварианты (слои, kp-*, FreezeGuard, RBAC)
- **lock/** — FreezeGuard: какие файлы можно менять без approval @chief-architect
- **rules/** — ui-manifest, encoding-windows, audit-checklist
- **audit/** — backlog, UI-план, отчёты TEAM-WORKFLOW
- **AI_CONTEXT.md** — сжатый контекст для агентов

**Роль:** точность для машин и pre-commit; не заменяет красивый UI для человека.

### YouGile — пульт для людей (PLM roadmap)

- Доска «Статус реализации» — категории 01–08 на колонке «План развития»
- EPIC с тегами `#CRM`, `#DOCS`, `#PLM`
- Иерархия через `subtasks` (не `parentTaskId`)
- Реестр `PLM-xxx` ↔ UUID для скриптов и ссылок в чате

**Роль:** «что строим», % по EPIC, видимость для команды и заказчика.

### Gap — интеграция не закрыта

```
Задумка:     YouGile (человек)  ←—— мост ——→  .opencode/ (AI + код)
Реальность:  YouGile (PLM)       ✗              .opencode/ (backlog, UI, freeze)
             sync-server (только % EPIC, локально)
```

## 4. Цель команды (формулировка из диалога)

### Для человека (владелец / PM / команда)

- Открыть **красивый интерфейс** (не markdown в IDE)
- Видеть **одним взглядом:** что сделано · что осталось · что требуется
- Работать **из дома** так же, как в офисе — без `start.ps1`, портов 3002, путей `D:\...`
- Не гадать, «куда делись вчерашние фиксы»

### Для AI (Cursor, OpenCode, агенты)

- **Один язык** с человеком: та же задача, то же название, тот же статус
- Читать **машиночитаемый** контекст в репо (не выдумывать backlog)
- Соблюдать **FreezeGuard** и AGENTS.md независимо от доски
- Участвовать в проекте **без** обязательного доступа к YouGile API в каждой сессии

### Ограничения (явные пожелания)

- **Без лишних усложнений** — не строить «второй Jira» в репо
- **Без жёстких привязок** к локальным путям и sibling-репозиториям для ежедневной работы
- **Довести задуманное** или **упростить сознательно** — но не держать половину системы «на всякий случай»

### В0. Текущее («как есть» — dual truth)

**Описание:** Статус проекта размазан: PLM и часть чеклистов — в YouGile; оперативный backlog и UI-план — только в `.opencode/audit/`; % EPIC — если запущен `yougile-sync-server` локально. Новые планы пишутся в md без обязательных карточек. AI читает AGENTS + md; человек смотрит YouGile — и видит разные картины.

| Критерий | Оценка |
|----------|--------|
| **Плюсы** | Ничего менять не нужно; freeze и инварианты уже работают; PLM-дерево частично живое |
| **Минусы** | Две правды; UI-работы «невидимы»; фрустрация; sync-server бесполезен без локального запуска |
| **Сложность** | Низкая (статус-кво) |
| **Локальный сервер** | Опционально для % EPIC |
| **Из дома** | YouGile ✅; полная картина ❌ (md только в репо) |
| **Роль ИИ** | Читает md + freeze; статус задач угадывает или спрашивает |
| **Роль человека** | YouGile для PLM; backlog — открывать файлы в IDE/GitHub |

### В1. Минимальный — YouGile единственный статус

**Описание:** Одна доска **«KPPDF — сейчас»** с колонками: **Сделано · В работе · Дальше · Блокеры**. Крупные темы — **метки** (UI, PLM, Bug), не отдельные вселенные. Markdown (`CHECKLIST-BACKLOG.md`, `UI-CONSISTENCY-PLAN.md`) — **справочник «что делать»** (один раз перенесли в карточки — живут в YouGile). Синхронизаторы **не используются**. ИИ получает контекст **ссылкой или названием карточки** в начале чата. Freeze / AGENTS — по-прежнему только в репо.

| Критерий | Оценка |
|----------|--------|
| **Плюсы** | Максимально просто; работает из дома (браузер); один статус для человека; нет localhost:3002 |
| **Минусы** | ИИ не видит статус без вашей фразы/ссылки; md может устареть как спецификация; PLM-дерево отдельно |
| **Сложность** | **Низкая** |
| **Локальный сервер** | **Не нужен** |
| **Из дома** | **✅ Полностью** |
| **Роль ИИ** | Работает по названию карточки + AGENTS + freeze; не читает YouGile API |
| **Роль человека** | Единственный пульт — YouGile; утром открыл доску — видишь всё |

**Первый шаг:** за полчаса вручную создать 10–15 карточек из UI-плана и backlog.

### В2. YouGile пульт + ручная дисциплина (3 доски)

**Описание:** Три доски в YouGile:
1. **PLM / Продукт** — существующая «Статус реализации» (01–08, EPIC, %)
2. **Оперативный Backlog** — P0–P2 из `CHECKLIST-BACKLOG.md`
3. **UI Kit** — колонки P0 / P1 / P2 из `UI-CONSISTENCY-PLAN.md`

**Жёсткое правило:** *«Нет карточки в YouGile = нет задачи»* — новый md-план без карточек в тот же день не принимается. Закрыли в коде → перетащили карточку / отметили чеклист в YouGile. Опционально: sync-server **только** для авто-% EPIC на доске PLM (не для backlog).

| Критерий | Оценка |
|----------|--------|
| **Плюсы** | Структура для зрелой команды; PLM отделён от ops; UI-план виден человеку |
| **Минусы** | Три места смотреть; дисциплина на людях; ИИ всё ещё без авто-снимка |
| **Сложность** | **Средняя** (организационная) |
| **Локальный сервер** | Опционально (только % EPIC) |
| **Из дома** | **✅** (YouGile в браузере) |
| **Роль ИИ** | Карточка + md-спецификация; freeze отдельно |
| **Роль человека** | Владелец досок; следит за правилом «нет карточки = нет задачи» |

### В3. YouGile пульт + авто-снимок в репо

**Описание:** YouGile остаётся **единственным местом смены статуса** (как В1 или В2). Добавляется **автоматический экспорт** в репо:

- Файл `.opencode/yougile-snapshot.yaml` — id, название, колонка, метки, %, ссылка, дата обновления
- Опционально: блок «Status from YouGile» в `CHECKLIST-BACKLOG.md` (только read-only, генерируется скриптом)
- Запуск: cron на CI, GitHub Action, или `tools/yougile-export-snapshot.js` по кнопке / раз в N часов
- В `AI_CONTEXT.md`: *«Статус задач — yougile-snapshot.yaml; не выдумывай»*

| Критерий | Оценка |
|----------|--------|
| **Плюсы** | **Один язык** для ИИ и человека; ИИ не нужен API YouGile в чате; git history снимков |
| **Минусы** | Нужен токен + job (CI или cron); задержка N минут; первоначальная разработка скрипта |
| **Сложность** | **Средняя** (техническая) |
| **Локальный сервер** | **Не обязателен** (лучше CI/cloud cron) |
| **Из дома** | **✅** человек; **✅** ИИ читает snapshot из репо |
| **Роль ИИ** | Читает snapshot + спецификации md + freeze |
| **Роль человека** | Меняет статус только в YouGile; не трогает snapshot руками |

### В4. Репо канон + YouGile зеркало

**Описание:** **Источник правды — markdown/YAML в репо.** YouGile — **зеркало для людей**. Скрипты (паттерн `setup-documents-yougile-module.js`, `yougile-restructure.js`) **создают/обновляют** карточки из:
- `UI-CONSISTENCY-PLAN.md` → доска UI Kit
- `CHECKLIST-BACKLOG.md` → доска Backlog

Каждая задача в md имеет id (`UI-P0-03`, `BL-P1-07`) → карточка YouGile с ссылкой на файл/строку. Статус меняется **в репо** (чекбокс) → скрипт push в YouGile (колонка, completed).

| Критерий | Оценка |
|----------|--------|
| **Плюсы** | Идеально для AI-first; всё в git; review через PR; воспроизводимо |
| **Минусы** | Человеку неудобно менять статус (IDE, не drag-and-drop); нужны скрипты и id-дисциплина |
| **Сложность** | **Средняя–высокая** |
| **Локальный сервер** | Не нужен (скрипты по событию / CI) |
| **Из дома** | ⚠️ Человек может смотреть YouGile; **менять** статус — через репо или скрипт |
| **Роль ИИ** | Главный потребитель md; обновляет чекбоксы в PR |
| **Роль человека** | Смотрит YouGile; статус — через merge PR или ручной скрипт |

### В5. Расширить yougile-sync-server

**Описание:** Доработать sibling `yougile-sync-server`:
- Bidirectional sync (YouGile ↔ `mapping.yaml` ↔ md/yaml)
- Экспорт backlog / UI-плана в snapshot или обратно в YouGile
- Расширить `config/mapping.yaml` — не только EPIC-%, но и ops-задачи
- Деплой sync-server на VPS (не только localhost) — чтобы работало без `start.ps1`

| Критерий | Оценка |
|----------|--------|
| **Плюсы** | Переиспользует существующий код; уже умеет polling и % EPIC |
| **Минусы** | Sibling-репо и путь `D:\...`; сложность bidirectional; риск конфликтов; нужен хостинг |
| **Сложность** | **Высокая** |
| **Локальный сервер** | Да (или VPS-аналог) |
| **Из дома** | ✅ если задеплоен в облако; ❌ если только localhost |
| **Роль ИИ** | Читает экспорт из репо или mapping |
| **Роль человека** | YouGile UI; зависит от направления sync |

## 7. Рекомендации по ролям

### Владелец продукта / малая команда (1–3 человека)

→ **В1 Минимальный**

- Одна доска «KPPDF — сейчас»
- md = справочник, статус только в YouGile
- Без sync-server в ежедневной жизни
- В чат с AI: ссылка на карточку + «смотри AGENTS.md»

### Зрелая PLM + EPIC % + несколько направлений

→ **В2 + опционально sync-server только для %**

- Три доски (PLM / Backlog / UI Kit)
- Правило «нет карточки = нет задачи»
- sync-server на VPS **только** для авто-% EPIC — не для ops

### Heavy AI automation (много агентов, мало ручного чата)

→ **В3** (YouGile + snapshot) **или В4** (репо канон + push в YouGile)

- В3 — если человек хочет drag-and-drop
- В4 — если PR и git — главный workflow

### Не рекомендуется без явного решения

- **В0** — сохраняет текущую боль
- **В5** — если нет ресурса на поддержку sibling-сервера
- **В6** — только если YouGile принципиально не подходит

## 8. Что прекратить делать (общее для всех вариантов)

1. **Писать новые md-планы без карточки в YouGile** (или без id в едином PROJECT-STATUS — для В7)
2. **Полагаться на localhost:3002** для ежедневного статуса — это dev-утилита, не пульт
3. **Ожидать, что все чеклисты уже в YouGile** — сегодня там только часть PLM (в основном 08 Документы)
4. **Дублировать статус** в md и YouGile с разными галочками — один источник правды для **статуса**, md — для **спецификации**
5. **Считать `project-readiness.yaml` UI для человека** — `enabled: false`, это file-only для AI
6. **Путать FreezeGuard и backlog** — freeze = «можно ли менять файл», не «сделана ли фича»
